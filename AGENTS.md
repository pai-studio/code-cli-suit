# 代码编写规范

## 配置
代码设计文档目录: docs/design
计划文档和结果目录: docs/dev
是否提交docs/目录: 是
git commit语言: 中文
文档语言: 中文
注释语言: 中文

## 如何确定下一步的工作
如果用户没有指定，应该思考本系统的最终目标是什么，当前代码状态下的高价值、最紧急的任务是什么。

## 如何做设计
对于设计，要参考api-design-zh这个skill。特别是用户接口，要提出不同方案，比较优缺点，然后综合方案才能定稿，然后这个定稿也要做2~3次查漏补缺确定，要标注必要的设计理由，将来更新时，用于假设不存在时的推翻，假设存在时防止被错误推翻。
设计命名为 {topic}-{date:YYYYMMDD}-{tag}.md 如，memory-20260606-overview.md

## 如何制定计划
计划文件命名为 PLAN-{XXX:数字}-{tag}.md 放在对应目录，比如命名为: PLAN-001-function.md

## 如何实施代码
按计划实施代码，如果计划实施前有更新，先更新计划文件再实施。实施完代码应该review一遍代码，如果有问题，应该修复，然后生成结果文件，比如PLAN-001-function-OUTCOME.md

## 如何提交代码
遵守我们的配置，如果提交docs/ ，应该在commit消息中记录我们实施的计划文件和结果文件，逐条详细记录我们的变更。commit后要立刻push。
